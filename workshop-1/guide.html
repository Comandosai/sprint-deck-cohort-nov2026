Rescue · Если SSH отвалился после Промпта 4
kex_exchange_identification: Connection closed

Симптом — при попытке подключиться SSH сразу обрывается:

kex_exchange_identification: Connection closed by remote host
Connection closed by <IP> port 22

Это обычно НЕ проблема SSH-ключа. Сервер закрыл соединение ещё до handshake. Частые причины:

  • • fail2ban забанил твой IP после серии попыток входа
  • • ufw limit 22/tcp сработал из-за залпа SSH-команд (часто — агент циклил su - clawd или повторные проверки root SSH)
  • • Меняется внешний IP (VPN / мобильный интернет)
  • • sshd слушает другой порт или перезапущен с ошибкой
⛔
НЕ продолжай долбить SSH агентом. Каждая попытка усиливает бан. Иди диагностировать через web-console / VNC / console провайдера VPS (Beget / Hetzner / etc.).

1. В web-console провайдера выполни:

web-console VPS
sudo systemctl status ssh --no-pager
sudo journalctl -u ssh -n 80 --no-pager
sudo fail2ban-client status sshd || true
sudo ufw status numbered
sudo ss -tlnp | grep ssh

2. На Mac узнай свой текущий внешний IP:

curl -4 ifconfig.me

3. Если IP в бане — разбань (в web-console VPS):

web-console VPS
sudo fail2ban-client set sshd unbanip <ТВОЙ_IP>
sudo systemctl restart fail2ban

4. Проверь обычный SSH из терминала Mac (не через агента):

ssh -vvv -i ~/.ssh/clawd_ed25519 clawd@<VPS_IP>

5. Если снова рвёт — временно (только для диагностики) останови fail2ban:

web-console VPS
sudo systemctl stop fail2ban
# после восстановления доступа — ОБЯЗАТЕЛЬНО включи обратно:
sudo systemctl start fail2ban
sudo systemctl is-active fail2ban
⚠️
Чего НЕ делать:
  • • не добавлять IP участника в ignoreip по умолчанию
  • • не ограничивать SSH «только с моего IP» (мобильный/VPN меняют его)
  • • не пересоздавать SSH-ключ и не переустанавливать VPS — это редко настоящая причина
  • • не отключать fail2ban навсегда
  • • после включения fail2ban / ufw limit — не пускать параллельные SSH-команды и не циклить проверки root-входа

Критерии стандарта A.5 (ufw active с rate-limit) и A.6 (fail2ban active) остаются обязательными — этот блок только про восстановление доступа, не про отключение защиты.

Выбери NEW_PORT в диапазоне 10000-60000: NEW_PORT=$((RANDOM % 50000 + 10000)) echo "NEW_PORT=$NEW_PORT" 2. СНАЧАЛА открой NEW_PORT в ufw (на случай если sshd перезагрузится): sudo ufw allow $NEW_PORT/tcp sudo ufw status 3. ПОТОМ правь sshd_config с бэкапом: sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.bak.$(date +%F) sudo sed -i "s/^#*Port 22/Port $NEW_PORT/" /etc/ssh/sshd_config sudo sed -i 's/^#*PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config sudo sed -i 's/^#*PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config echo "MaxStartups 5:30:10" | sudo tee -a /etc/ssh/sshd_config echo "MaxSessions 5" | sudo tee -a /etc/ssh/sshd_config 4. sudo sshd -t (проверка синтаксиса). Если OK: sudo systemctl reload sshd 5. СКАЖИ МНЕ СТОП. Я открою второе окно терминала на Mac и проверю: ssh -i ~/.ssh/clawd_ed25519 -p $NEW_PORT clawd@$VPS_IP "echo OK" Скажу тебе работает или нет. 6. Если работает — закрывай 22: sudo ufw delete allow 22/tcp sudo ufw status 7. Подтяни fail2ban (bantime 24h): sudo tee -a /etc/fail2ban/jail.local <<EOF [sshd] enabled = true port = $NEW_PORT bantime = 86400 findtime = 600 maxretry = 3 EOF sudo systemctl restart fail2ban sudo fail2ban-client status sshd 8.
