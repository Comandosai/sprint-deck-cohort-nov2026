# Блок 15: Mem0 + Qdrant — оптимизация памяти для CRM-функций

> **Что:** Подключение плагина `@mem0/openclaw-mem0` к OpenClaw, развёртывание локального стека (Qdrant + Ollama) и настройка авто-захвата/авто-вспоминания фактов о клиентах.
> **Зачем:** Бот должен помнить каждого человека, хранить факты, историю общения и подставлять контекст в промпт перед каждым ответом — без этого CRM-функция не работает.
> **Время:** 1.5–2.5 часа (с проверкой `/reset` и тестом recall).

---

## 🎯 Цель блока

К концу блока:

1. Установлен официальный плагин `@mem0/openclaw-mem0` (npm), привязан к слоту `memory` в `openclaw.json`.
2. Локально запущен Qdrant в Docker с persistent-volume (порт 6333 REST + 6334 gRPC, dashboard на `/dashboard`).
3. Запущен Ollama с embedding-моделью (`nomic-embed-text` или `mxbai-embed-large`) — нулевая стоимость, либо альтернатива через `text-embedding-3-small` ($0.02/1M токенов).
4. `autoCapture: true` — после каждого turn-а агент извлекает факты (имена, роли, проекты, предпочтения) и пишет в Qdrant.
5. `autoRecall: true` — перед каждым ответом происходит инъекция top-K релевантных воспоминаний в working context (бюджет 1500 токенов по умолчанию).
6. Прошёл тест: «расскажи факт → `/reset` → спроси о факте → факт всплывает».
7. Чёткое разделение: **MEMORY.md (Блок 6) = декларативные правила/предпочтения; Mem0 = семантический CRM-стор (фактов о людях, событий, истории).**

### Что чему соответствует в OpenClaw

- **Слот `memory`** в `openclaw.json` — единственное место, где регистрируется memory-плагин. Подключение нескольких плагинов одновременно в один слот не предусмотрено.

## 💡 Лайфхаки и про-приёмы

### 1. Ставьте `searchThreshold` низким, а `topK` — умеренным
По умолчанию `topK: 5`, `searchThreshold: 0.1` (или 0.3 — `[VERIFY]` зависит от версии). Для CRM лучше `topK: 8` + `searchThreshold: 0.25`. Ниже 0.2 в recall начинает «протекать» мусор; выше 0.4 — теряются непрямые связи («Иван» vs «Ваня»).

### 2. Включайте `keywordSearch: true` всегда
Embeddings плохо ловят точные ID, телефоны, ИНН, GitHub-логины. Гибрид (вектор + keyword) бесплатен по latency, но ловит больше. В конфиге recall-скилла: `"keywordSearch": true`.

### 3. `identityAlwaysInclude: true` — must для CRM
Гарантирует, что имя/роль текущего собеседника всегда в контексте, даже если recall-скоры низкие. Без этого бот может «забыть, с кем говорит» в длинном диалоге.

### 4. Выделите `userId` на каждого реального человека
Не используйте OS-username (дефолт). Для каждого клиента/собеседника — отдельный `userId` (например `tg:123456789` для Telegram). Тогда `memory_search` не смешает воспоминания о Васе и Пете. Это и есть multi-user mem0.

### 5. Mem0 vs MEMORY.md — чёткая граница
- **MEMORY.md** (Блок 6): декларативные правила бота («отвечай на ты», «не используй emoji», «всегда подписывайся как Дима»). Источник истины — Markdown-файл, виден человеку, версионируется в git.
- **Mem0**: факты о третьих лицах, история разговоров, события, предпочтения, динамические данные. Источник истины — Qdrant. Не редактируется руками.
- **Конфликт?** Если `triage` пытается записать в Mem0 факт «бот должен отвечать на ты» — это утечка. В системный промпт добавьте: *«Не сохраняй в долгую память правила собственного поведения — они в MEMORY.md».*

### 6. Persistent-volume для Qdrant — не пропускайте
Без `-v "$(pwd)/qdrant_storage:/qdrant/storage:z"` все воспоминания испаряются при `docker restart`. Бэкап = регулярно `tar czf qdrant_$(date +%F).tgz qdrant_storage/`. Восстановление: stop контейнер → распаковать → start.

### 7. Ollama в native-режиме на VPS, не в Docker
На VPS Linux `curl https://ollama.ai/install.sh | sh` запускает сервис нативно — меньше накладных расходов, прямой доступ к /dev/dri (если есть GPU). Docker-контейнер Ollama добавляет лишний слой и часто ломает GPU-проброс. На macOS-разработке — пакет `Ollama.app`.

### 8. Embedding-модель должна совпадать на запись и чтение
Если поменять `nomic-embed-text` (768 dim) на `text-embedding-3-small` (1536 dim) — старая коллекция Qdrant **сломается** (несовпадение размерности). Решение: при смене embedder-а — **дроп коллекции** (`memory_*` collection в Qdrant) и переэкстракция из истории. Не мигрируется автоматически.

### 9. `dream` skill включайте только если бот живёт >1 недели
Скилл идёт фоном, склеивает дубликаты («работает в Acme» + «сотрудник Acme» → один факт). На свежей установке — шум; включайте после 50–100 разговоров.

### 10. Reranking — нужен только если topK>10
Mem0 уже делает re-rank (`"rerank": true` в recall) внутри плагина. Внешний reranker (Cohere rerank-3, BGE-reranker) добавляет +200–500 ms. Для CRM с topK=5–8 — лишнее.

### 11. GDPR-готовность: `memory_delete` + аудит
Включите логирование всех `memory_add` / `memory_delete` (лог-файл в workspace). На запрос «удалите мои данные» — `openclaw mem0 search "<имя>" --user-id <id>` → пройти по результатам и `memory_delete`. Полное удаление всего пользователя: дроп userId-фильтрованных точек в Qdrant scroll-API.

### 12. Cost-bound: оцените трафик заранее
- 1 диалог ≈ 800 input tokens (история) + 300 на extraction prompt + 50 на embedding = ~1150 токенов на extraction (если LLM — облачный).
- Embeddings на запись: ~5–10 фактов × 30 токенов = 150–300 токенов на диалог.
- 100 диалогов = ~115K LLM tokens + ~25K embedding tokens = на `text-embedding-3-small` это **$0.0005 за embeddings** (копейки). Дорого только LLM-часть extraction. На Ollama локально = 0.

### 13. `MEM0_TELEMETRY=false` обязательно
Mem0 шлёт анонимную телеметрию по умолчанию. Для CRM с персданными — отключайте до старта.

### 14. Ставьте `tokenBudget` recall-а в 1000–1800
Дефолт 1500 — норм. Меньше 1000 → recall обрезает важный контекст. Больше 2000 → раздувает каждый turn, дорого + Claude/GPT может «потерять» свежий вопрос пользователя в шуме.

## 📋 Готовые команды и конфиги

### docker-compose.yml для Qdrant + Ollama

Положить в `~/openclaw/infra/memory/docker-compose.yml`:

```yaml
services:
  qdrant:
    image: qdrant/qdrant:latest
    container_name: openclaw-qdrant
    restart: unless-stopped
    ports:
      - "6333:6333"   # REST API + dashboard на /dashboard
      - "6334:6334"   # gRPC
    volumes:
      - ./qdrant_storage:/qdrant/storage:z
    environment:
      - QDRANT__SERVICE__API_KEY=${QDRANT_API_KEY:-}   # пусто = без auth (только за фаерволом!)
      - QDRANT__LOG_LEVEL=INFO
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:6333/readyz"]
      interval: 10s
      timeout: 3s
      retries: 5

  ollama:
    image: ollama/ollama:latest
    container_name: openclaw-ollama
    restart: unless-stopped
    ports:
      - "11434:11434"
    volumes:
      - ./ollama_models:/root/.ollama
    # На VPS без GPU — оставляем как есть.
    # На машине с NVIDIA: раскомментировать
    # deploy:
    #   resources:
    #     reservations:
    #       devices:
    #         - driver: nvidia
    #           count: 1
    #           capabilities: [gpu]
```

Запуск:

```bash
cd ~/openclaw/infra/memory
docker compose up -d
# Дождаться, пока Qdrant ответит:
curl -s http://localhost:6333/readyz
# Скачать embedding-модель (один раз):
docker exec openclaw-ollama ollama pull nomic-embed-text
# Проверить:
curl http://localhost:11434/api/embeddings -d '{"model":"nomic-embed-text","prompt":"hello"}'
```

> **Замечание по сети:** на VPS обязательно закрыть порты 6333/6334/11434 в `ufw` (`ufw deny 6333`) — Qdrant и Ollama по дефолту без auth. Доступ — только из контейнера OpenClaw по `localhost` или внутренней docker-сети.

## ⚠️ Подводные камни

1. **Node 25 + OSS-режим = падение** (issue **mem0ai/mem0#4156**, https://github.com/mem0ai/mem0/issues/4156 — sqlite3 binding под jiti). На момент верификации issue закрыт через PR #4270, но до подтверждения попадания фикса в вашу версию плагина — сидите на Node 22 LTS. **Не путать с `openclaw/openclaw#4156`** — это совсем другой баг (WhatsApp TTS / MP3 vs Opus).
2. **Смена embedder-а ломает коллекцию.** Размерности 768/1024/1536/3072 несовместимы. Только дроп коллекции и реиндекс.
3. **Qdrant без auth открыт в интернет** — катастрофа с персданными. Проверить `ufw status`, привязать к `127.0.0.1` если запущено вне docker-network.
4. **`autoCapture` извлекает мусор** в первые дни. Просматривайте `openclaw mem0 list` раз в день, удаляйте «бот должен делать X» — это не CRM-факты, это поведение.
5. **`/reset` чистит ТОЛЬКО разговор, НЕ Mem0.** Это и есть тест recall — но если хочется реально стереть, нужен `memory_delete` или дроп Qdrant-коллекции.
6. **Конфликт фактов**: «Иван — CTO» vs «Иван — CEO». Без `dream`-скилла оба остаются. Mem0 OSS не строит граф (на момент апреля 2026 — `[VERIFY]`). Графовый режим — только в Mem0 Platform/Cloud через graph_store.
7. **`searchThreshold` слишком низкий** = recall тянет всё подряд → каждый ответ дороже на 1500+ токенов LLM.
8. **`tokenBudget` слишком высокий** = большой turn → провайдер LLM начинает резать или дорого считать. Стандарт 1500.
9. **OpenAI ключ для extraction в Cloud-режиме**: если не задан — плагин падает на первой же auto-capture. В OSS-режиме можно полностью на Ollama.
10. **Telemetry**: `MEM0_TELEMETRY=false` ставится **до** старта OpenClaw. Если забыли — ребут процесса.
11. **Persistence Qdrant**: без `:z` SELinux на CentOS/RHEL VPS заблокирует запись в volume. На Ubuntu/Debian флаг безопасен.
12. **Ollama-модели качаются медленно** (270 MB–4 GB). На слабом VPS — 5–15 мин. Не таймаутить compose.

## 🧪 Верификация

### Тест 1 — базовый recall (обязательный)

```
Шаг 1. Написать боту: «Меня зовут Дмитрий, я работаю над спринтом OpenClaw,
       мой любимый язык — Python, дочку зовут Лиза, ей 6 лет.»
Шаг 2. Подождать 5 секунд (auto-capture).
Шаг 3. /reset (или новая сессия).
Шаг 4. Спросить: «Как зовут мою дочь и сколько ей лет?»
Ожидаемое: «Лизу зовут вашу дочь, ей 6 лет.»
```

Если факт не всплыл — проверить:

```bash
openclaw mem0 search "дочь Лиза" --user-id <ваш-id>
# Если результаты есть, а в ответе — нет: проверить tokenBudget recall, identityAlwaysInclude
# Если результатов нет: triage не сохранил → проверить logs OpenClaw, autoCapture=true
```

- **ДО (Блок 6 — память/MEMORY.md):** граница «MEMORY.md = декларативные правила для самого бота, Mem0 = семантический CRM-стор фактов о третьих лицах и истории». В Блоке 6 в `MEMORY.md` добавьте секцию «# Memory routing» с этой границей — это инструктирует triage-skill, что не сохранять.
- **ДО (Блок 1 — VPS):** в `ufw` закройте 6333/6334/11434. Outbound — только если идёте Cloud-режимом или OpenAI embeddings.
- **ДО (Блок 3 — LLM-провайдеры):** в OSS-режиме можно использовать ту же Ollama (`llama3.1:8b` или `qwen2.5:7b`) для extraction — экономия на LLM-токенах. В cloud-режиме нужен `OPENAI_API_KEY` для `gpt-5-mini` (extraction).
- **ПОСЛЕ (Блок 20 — финальная конфигурация):** мониторинг Qdrant в дашборде (Блок 13), бэкап-скрипт в общую инфра-папку, Mem0-метрики в общий health-check.
