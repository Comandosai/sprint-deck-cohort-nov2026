# Блок 16: Мульти-агенты (команда сотрудников)

> **Что:** превратить одного «универсального солдата» в команду из 4–7 специализированных агентов (main / research / content / support / ops / finance / social), каждый со своим workspace, SOUL.md, моделью и Telegram-топиком. Маршрутизация — через `bindings`, делегирование — через `sessions_spawn`.
> **Зачем:** один агент с раздутым контекстом стоит дороже и тупит чаще, чем 4 специализированных. Дешёвый research на Sonnet идёт 8 минут параллельно, пока main на Opus занят чем-то ещё. Telegram Forum Supergroup — один топик = один сотрудник, в одном чате со всеми.
> **Время:** 2–3 часа на базовую 4-агентную команду; +1 час на каждого дополнительного «сотрудника».

---

## 🎯 Цель блока

К концу блока у Дмитрия должно быть:

1. `openclaw.json` с описанием 4 именованных агентов — `main`, `research`, `content`, `support` — и (по желанию) `ops`, `finance`, `social`.
2. Раздельные workspace-директории `~/.openclaw/workspace-<id>/` с собственным `SOUL.md`, `AGENTS.md`, `USER.md`, `IDENTITY.md`.
3. Раздельные `agentDir` в `~/.openclaw/agents/<id>/agent` (НИКОГДА не пересекаются — это чёткое предупреждение из доков).
4. Telegram Forum Supergroup с топиками `#research`, `#content`, `#support` (+ `#ops`, `#finance`, `#social` для расширенной команды) и `#general` под `main`.
5. Bindings, маршрутизирующие сообщения по `accountId` / `peer` (плюс topic-роутинг через workaround — см. ниже).
6. Рабочее делегирование: пишешь в `#general` → `main` зовёт `sessions_spawn(agentId: "research", task: "...")` → research отчитывается обратно в `#general`.
7. Каждый агент — на своей модели (cost control + специализация): research на Sonnet, content на Kimi K2.5/Opus, support на DeepSeek V3, main на Opus.
8. Per-agent spending caps и audit log делегирований.

## 🛠️ Конкретные инструменты и версии

### Из официальной документации OpenClaw (verified)

- **CLI команды:**
  - `openclaw agents add <id>` — создать агента
  - `openclaw agents list --bindings` — проверить маршрутизацию
  - `openclaw channels status --probe` — диагностика каналов
  - `openclaw gateway restart` — рестарт gateway после правки конфига
  - `openclaw channels login --channel telegram --account <name>` — авторизация второго бота
- **Slash-команды внутри сессии:**
  - `/subagents list` — кто сейчас работает
  - `/subagents kill <id|#|all>` — прибить (или всех)
  - `/subagents log <id|#> [limit] [tools]` — логи sub-agent
  - `/subagents info <id|#>` — метаданные
  - `/subagents send <id|#> <message>` — отправить сообщение работающему sub-agent
  - `/subagents steer <id|#> <message>` — корректировать ход (steering)
  - `/subagents spawn <agentId> <task> [--model <m>] [--thinking <level>]` — ручной спавн
  - `/focus <subagent-label|session-key>` / `/unfocus` — переключиться в сессию sub-agent
  - `/agents` — список и переключение между основными агентами
  - `/stop` — прервать requester + всех его детей и внуков
- **Инструмент `sessions_spawn`** — главный механизм делегирования. Параметры:
  - **обязательный:** `task` (string)
  - **опциональные:** `label`, `agentId` (по умолчанию = requester), `model`, `thinking`, `runTimeoutSeconds` (0 = без лимита), `thread` (для Discord-потоков), `mode` (`"run"` / `"session"`), `cleanup` (`"delete"` / `"keep"`), `sandbox` (`"inherit"` / `"require"`), `context` (`"isolated"` / `"fork"`), `runtime` (`"subagent"` / `"acp"`)
- **Уровни вложенности sub-agents:**
  - depth 0 = `agent:<id>:main` — главный
  - depth 1 = `agent:<id>:subagent:<uuid>` — sub-agent (он же мини-оркестратор, если `maxSpawnDepth >= 2`)
  - depth 2 = `agent:<id>:subagent:<uuid>:subagent:<uuid>` — worker, спавнить дальше нельзя
  - максимум — depth 5, **рекомендовано 2**

### Файлы workspace агента

В `~/.openclaw/workspace-<id>/` находятся:
- `SOUL.md` — персона и стиль (тон, юмор, прямолинейность) — Блок 5
- `AGENTS.md` — операционные правила и приоритеты
- `USER.md` — что агент знает о Дмитрии
- `IDENTITY.md` — имя и характер
- `MEMORY.md` (опц.) — долгосрочная память
- `memory/YYYY-MM-DD.md` — ежедневные логи
- `TOOLS.md`, `HEARTBEAT.md`, `BOOTSTRAP.md`
- `skills/`, `canvas/` — папки

### Sub-agent токены и инструменты (verified)

Sub-agent **по умолчанию НЕ получает**: `sessions_list`, `sessions_history`, `sessions_send`, `sessions_spawn`. То есть лист worker не может звать ещё одного. Оркестратор (depth 1 + `maxSpawnDepth >= 2`) — получает эти инструменты обратно.

### Сравнение с другими фреймворками (для понимания паттерна)

- **CrewAI** — role-based crews, hierarchical process с Manager Agent (Opus/GPT-5) и Worker Agents (Haiku/cheap). Token overhead ~18% против LangGraph на тех же задачах.
- **LangGraph** — directed graph с conditional edges, audit trails, rollback. Production-grade; обогнал CrewAI по звёздам в начале 2026.
- **AutoGen / AG2** — conversational GroupChat (агенты «обсуждают»).
- **Google ADK** — «org chart, not group chat», hierarchical tree, корневой агент делегирует вниз.
- **A2A Protocol v1.2** — стандарт межагентной связи поверх HTTP + SSE + JSON-RPC. Дополняет MCP (MCP = агент↔инструмент, A2A = агент↔агент).

OpenClaw `sessions_spawn` ближе всего к ADK / CrewAI hierarchical: один main делегирует один-в-один, push-based announcement.

## 💡 Лайфхаки и про-приёмы

### 1. `context: "isolated"` по умолчанию, `"fork"` — только когда правда нужно

`isolated` (свежая чистая сессия) — экономит токены и не тащит шум главной беседы. `fork` копирует весь транскрипт requester'а — это дорого и обычно не нужно. Доки прямо говорят: «Use rarely» про fork. Включай fork только если research должен видеть, ЧТО уже обсуждалось, не пересказывая.

### 2. Sub-agents сажай на дешёвую модель глобально

```json5
{
  agents: {
    defaults: {
      subagents: { model: "claude-3-5-haiku-20241022" }
    }
  }
}
```

Это рекомендация из официальной доки: основной агент остаётся на Opus, а вся фоновая «грязная» работа — Haiku/Sonnet/DeepSeek. Сразу −60–80% бюджета на типовых задачах. [VERIFY конкретного коэффициента под твой workload]

### 3. Push-based — НИКОГДА не полить `sessions_list` в цикле

Это прямо вредный паттерн (доки явно предупреждают). Sub-agent сам объявит результат, когда закончит — через push-based announcement в канал requester. Если ты в цикле зовёшь `sessions_list` или `sessions_history` — ты тратишь токены и порождаешь race conditions.

Для оркестрации читай результат через `sessions_history` (а не сырой файл транскрипта): он нормализован, redacted (credentials скрыты), большие блоки обрезаны.

### 4. `maxSpawnDepth: 2` — золотая середина

`1` — слишком плоско, main всегда сам должен решать. `3+` — research спавнит research, который спавнит research, и ты теряешь контроль. `2` — main делегирует research, research может разбить задачу на 2–3 параллельных worker'а, и всё.

### 5. Per-agent spending cap через model selection (нет встроенного $-лимита, но есть селектор модели)

OpenClaw не имеет hard-cap «$5/день на agent» из коробки [VERIFY], но через `model:` per agent ты ставишь cost-ceiling по архитектуре:
- `main` = Opus 4.7 (дорого, но редко)
- `research` = Sonnet 4.6 (золото cost/quality)
- `content` = Kimi K2.5 / Sonnet (дешевле, креатив)
- `support` = DeepSeek V3 / Haiku (дёшево, FAQ)
- `ops`/`finance` = Haiku (служебка)

Дополнительно — `runTimeoutSeconds: 900` (15 мин) на sub-agents в defaults, чтобы зависшая задача не жрала бюджет.

### 6. Workaround для Telegram per-topic (фича не реализована)

Поскольку `bindings` НЕ матчит `topicId`, делай **multi-account workaround**: одна Forum Supergroup, но **разные боты** на каждый топик. У каждого топика — отдельный bot token, и `bindings` матчат `accountId` бота. Это коряво, но работает сегодня.

Альтернатива: один бот, один общий чат, и main парсит топик из метаданных и сам зовёт `sessions_spawn(agentId: ...)` через скилл-роутер. Менее надёжно, требует кастомного скилла, но не требует N ботов.

### 7. Никогда не переиспользуй `agentDir` — это #1 правило

Из доков прямо: «Никогда не переиспользуйте `agentDir` между разными агентами — вызовет коллизии аутентификации и сессий.» Каждый агент = свой `~/.openclaw/agents/<id>/agent`. OAuth-токены не клонируются — авторизуй каждого отдельно.

### 8. Конфликты «два агента пишут в один файл» — изолируй workspace-ами

OpenClaw НЕ имеет встроенного file-lock между агентами [VERIFY — в доках не упомянут]. Решение архитектурное: каждый workspace = свой каталог. Если нужна общая память — выноси в QMD `extraCollections` (cross-search), а не общий writable файл.

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        qmd: {
          extraCollections: [
            { path: "~/agents/research/sessions", name: "research-sessions" }
          ]
        }
      }
    }
  }
}
```

Это позволяет main «видеть» что делал research, но не писать туда.

### 9. Аудит делегирования через session naming

Каждая sub-agent сессия именуется как `agent:<id>:subagent:<uuid>`. Делай ежедневный cron, который грепает `~/.openclaw/agents/*/sessions/` за вчера и собирает: кто делегировал → кому → с какой задачей → сколько токенов → какой результат. Это твой audit log без дополнительных тулов.

### 10. Failure recovery — `archiveAfterMinutes` + `runTimeoutSeconds`

```json5
{
  agents: {
    defaults: {
      subagents: {
        runTimeoutSeconds: 900,
        archiveAfterMinutes: 60,
        maxConcurrent: 8
      }
    }
  }
}
```

`runTimeoutSeconds: 900` — sub-agent умирает через 15 минут, не сжигая бюджет. `archiveAfterMinutes: 60` — архивируется через час. `maxConcurrent: 8` — не больше 8 одновременно (защита от каскадного DoS на API).

Failure recovery: если research упал, main должен поймать `Status: failed` в announcement и решить — retry с другим промптом, retry на другой модели, или вернуть пользователю «не получилось». Этому тренируется через `AGENTS.md` в workspace `main`.

### 11. Hierarchical для отчётов, peer-to-peer для коллабораций

- **Hierarchical (main → research):** для линейных запросов вида «найди X». Main выдаёт задачу, research отвечает. Чистый ответственностью.
- **Peer-to-peer (через `agent_send` или общий QMD):** когда content и research должны итеративно обмениваться. Менее структурированно, но эффективнее на креативных задачах. **[VERIFY: agent-send как отдельный инструмент в доках упомянут — `/tools/agent-send`]**

Правило большого пальца: для 80% задач hierarchical через `sessions_spawn` достаточно. Peer-to-peer — только если две головы реально лучше одной.

### 12. `mentionPatterns` для разделения в групповых чатах

В Forum Supergroup можно сделать одного бота с разными `mentionPatterns`:

```json5
{
  agents: {
    list: [
      { id: "research", groupChat: { mentionPatterns: ["@r", "@research"] } },
      { id: "content", groupChat: { mentionPatterns: ["@c", "@content"] } },
      { id: "support", groupChat: { mentionPatterns: ["@s", "@support"] } }
    ]
  }
}
```

Тогда `@r найди X` идёт в research, `@c напиши пост` — в content. Это проще чем N ботов, но требует, чтобы агенты были biased к одному channel/account и различались по mention. **[VERIFY совместимость с Telegram — точно работает в WhatsApp по доке семейного агента; для TG требует теста]**

### 13. Не давай sub-agents `gateway` и `cron` инструменты

```json5
{
  tools: {
    subagents: {
      tools: { deny: ["gateway", "cron"] }
    }
  }
}
```

Sub-agent не должен модифицировать gateway или ставить cron-задачи — это привилегия main. Иначе research спавнит cron, который спавнит ещё research'а, и через сутки у тебя 50 фоновых сессий.

### 14. Голосование/консенсус — простой паттерн

OpenClaw из коробки не имеет «голосования» [VERIFY]. Но можно сделать так: main для важного решения спавнит ТРИ research-sub-agent'а с одинаковой задачей и разными моделями (Opus, Sonnet, GPT-5). Затем main сам сравнивает три ответа. Это и есть «консенсус». Стоит дороже, но ловит галлюцинации.

### А. Создать 4 базовых агента

```bash
# main уже есть из бутстрапа, добавляем остальных
openclaw agents add research

# проверка
```

Каждая команда создаёт `~/.openclaw/agents/<id>/agent/` и (если запущен бутстрап) — стартовый workspace.

### Б. Полный `openclaw.json` для 4-агентной команды

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2,
        maxChildrenPerAgent: 5,
        maxConcurrent: 8,
        runTimeoutSeconds: 900,
        archiveAfterMinutes: 60,
        model: "claude-3-5-haiku-20241022"

            { path: "~/.openclaw/agents/content/sessions", name: "content-sessions" },
            { path: "~/.openclaw/agents/support/sessions", name: "support-sessions" }
          ]
        }
      }
    },
    list: [
      {
        id: "main",
        default: true,
        workspace: "~/.openclaw/workspace-main",
        agentDir: "~/.openclaw/agents/main/agent",
        model: "anthropic/claude-opus-4-7",

        groupChat: { mentionPatterns: ["@main", "@pm"] }

      },
      {
        id: "content",

    { agentId: "main", match: { channel: "telegram", accountId: "default" } }
  ],

  channels: {
    telegram: {
      enabled: true,
      accounts: {
        default: {
          botToken: "TOKEN_MAIN_BOT_FROM_BOTFATHER",
          dmPolicy: "allowlist",
          allowFrom: ["tg:USER_ID"]
        },
        research_bot: {
          botToken: "TOKEN_RESEARCH_BOT",
          dmPolicy: "allowlist",
          allowFrom: ["tg:USER_ID"]
        },
        content_bot: {
          botToken: "TOKEN_CONTENT_BOT",
          dmPolicy: "allowlist",
          allowFrom: ["tg:USER_ID"]
        },
        support_bot: {
          botToken: "TOKEN_SUPPORT_BOT",
          dmPolicy: "allowlist",
          allowFrom: ["tg:USER_ID"]
        }
      }
    }
  },

  tools: {
    subagents: {
      tools: {
        deny: ["gateway", "cron"]
      }
    }
  }

```

### В. SOUL.md — пример различий по агентам (Блок 5 расширение)

**`workspace-main/SOUL.md`** — PM/диспетчер:
```markdown
# main — Project Manager / Диспетчер

Ты — PM. Принимаешь запрос, решаешь: делать сам, или делегировать.
Если делегируешь — выбирай минимально достаточного исполнителя:
- research → факты, поиск, анализ источников
- content → текст для публикации
- support → ответ на типовой вопрос пользователя

Стиль: коротко, по делу, без воды. Не пересказывай ответ research — сразу выдавай суть и линкуй на full transcript.
Никогда не извиняйся за делегирование. Это твоя работа.
```

**`workspace-research/SOUL.md`** — research:
```markdown
# research — Аналитик

Ты ищешь факты. Любой вывод — со ссылкой. Если источник кривой — помечаешь [VERIFY].
Никаких «думаю», «возможно». Либо знаешь, либо не знаешь.
Стиль: bullet points, числа, даты. Никаких эмодзи.
```

**`workspace-content/SOUL.md`** — content:

```markdown
# content — Копирайтер

Ты пишешь текст для публикации. Hook первым предложением. Никаких «в современном мире».
Используешь активный залог, конкретные примеры, числа.
Длина — ровно та, что нужна. 280 символов — значит 280, не 350.
```

**`workspace-support/SOUL.md`** — support:

```markdown
# support — FAQ-агент

Ты отвечаешь пользователям на типовые вопросы. Дружелюбно, без воды.
Если вопрос сложный — пишешь "это не FAQ, передаю PM" и зовёшь main.
```

### Г. Telegram Forum Supergroup — пошагово

1. **Создай 4 ботов в @BotFather** (если идёшь по варианту A):
   - `@dim_main_bot`, `@dim_research_bot`, `@dim_content_bot`, `@dim_support_bot`
   - Получи 4 токена, положи в `openclaw.json` как выше.
   - Для каждого: `/setprivacy → Disable` (иначе бот не видит сообщения в группе).

2. **Создай Forum Supergroup в Telegram:**
   - Создаёшь обычную группу → Settings → Group Type → Supergroup → Topics → On.
   - Создаются топики: `General`, `#research`, `#content`, `#support`. (`General` обязателен и автоматический.)

3. **Добавь всех 4 ботов в группу:**
   - Каждому дай право Admin (минимум: Manage Topics + Send Messages).
   - В каждом топике пингани соответствующего бота, чтобы он «увидел» топик.

4. **Запусти gateway:**
   ```bash
   openclaw gateway restart
   openclaw channels status --probe
   ```

5. **Тест pairing:**
   - В DM каждому боту напиши `/start` — будет pairing code (если `dmPolicy: "pairing"`) или сразу примет (если `allowlist` и ты в нём).
   - Если allowlist — твой `tg:USER_ID` ищется так: пиши боту → `openclaw logs --follow` → находишь `from.id`.

6. **Тест делегирования** в `#general`:
   ```
   @main найди 3 источника про A2A protocol v1.2
   ```
   `main` должен ответить `[Спавню research...]` и через минуту прислать сводку. Параллельно в `#research` появится транскрипт sub-agent сессии (announce-сообщение).

> **Ограничение варианта А:** в каждом топике лучше «жить» одному боту, иначе другие боты видят чужие сообщения и могут отвечать. Лечится `dmPolicy`/`groupAllowFrom` + явным `requireMention: true` для каждого аккаунта. Но это уже кастомизация под канал.

### Д. Альтернатива — один бот + mention-роутинг (вариант Б)

Если не хочешь возиться с 4 ботами:

```json5
{
  channels: {
    telegram: {
      enabled: true,
      accounts: {
        default: {
          botToken: "ONE_BOT_TOKEN",
          dmPolicy: "allowlist",
          allowFrom: ["tg:USER_ID"]
        }
      }
    }
  },
  bindings: [
    { agentId: "main", match: { channel: "telegram", accountId: "default" } }
  ]
}
```

Маршрутизация по mention (`@research`, `@content`, ...) делается **скиллом** в `workspace-main/skills/route-mentions.md`, который main отрабатывает в каждом сообщении и зовёт `sessions_spawn(agentId: <matched>)`. Это менее автоматично, требует main всегда быть «на проводе».

### Е. Тест-сценарий: сложная задача → делегирование → отчёт

```
@main составь короткий обзор A2A protocol v1.2 для блог-поста.
1) research — найди 5 свежих источников за 2026
2) content — напиши пост 500 слов на основе сводки research
3) дай мне финальный текст
```

Ожидаемое поведение `main`:
1. Зовёт `sessions_spawn(agentId: "research", task: "5 источников по A2A v1.2 за 2026", model: "claude-sonnet-4-6", runTimeoutSeconds: 600)`.
2. Получает `Status: success`, читает транскрипт через `sessions_history`.
3. Зовёт `sessions_spawn(agentId: "content", task: "Пост 500 слов на основе [сводка от research]", context: "isolated")`.
4. Получает результат, форматирует и отдаёт пользователю.

Если research упал (`Status: failed` / `Status: timed out`):
- Main retry с другой моделью или меньшей задачей
- Если опять fail — честное «не получилось, вот что нашёл», без галлюцинаций

## ⚠️ Подводные камни

1. **Telegram per-topic routing НЕ работает** (issue #27562 открыт на апрель 2026). Не закладывайся на `topicId` в bindings — единственное надёжное решение пока: multi-account (N ботов) или mention-роутинг внутри main.
2. **Issue #4385: `/new` сбрасывает default-сессию.** Если в Telegram пишешь `/new` через не-default агента, default-сессия может сброситься. Workaround — явно `agentId` в команде. **[VERIFY текущий статус — мог быть пофикшен после открытия]**.
3. **`agentDir` НИКОГДА не разделяется между агентами** — пересечение = коллизии аутентификации и сессий. Это первое предупреждение в доках.
4. **OAuth токены не клонируются** — каждый агент авторизуется отдельно. Один Google-аккаунт на все агенты не получится без отдельного логина.
5. **`tools.elevated` — глобальный, основан на отправителе.** Не настраивается per-agent. Если sender в `elevated` — у него права на всех агентах.
6. **Sub-agent НЕ получает `sessions_spawn`** по умолчанию — то есть depth-2 worker не делегирует дальше. Это by design, не баг. Если нужно глубже — `maxSpawnDepth` + явная конфигурация инструментов.
7. **Push-based announcement — best-effort.** Если gateway упал между запуском sub-agent и завершением — announce может потеряться. Sub-agent отработает, но main не узнает. Защита: `runTimeoutSeconds` + периодическая проверка через `/subagents list` (НЕ в цикле!).
8. **Workspace ≠ песочница.** По умолчанию абсолютные пути могут выходить за её границы. Для реальной изоляции — `agents.defaults.sandbox: { mode: "all", scope: "agent" }` (Docker). Это медленнее, но безопасно для семейного/публичного агента.
9. **`memorySearch.qmd.extraCollections` даёт READ-only cross-search.** Не пытайся писать в чужой workspace через QMD — это для поиска, не для записи. Для общих данных — отдельный shared-каталог + явный pull через скилл.
10. **Cost overruns на Opus в main.** Главный агент на Opus 4.7 — это самый дорогой узел. Если main делегирует мало и сам делает много — счёт вырастет в 5–10x. Лекарство: жёсткий SOUL.md «делегируй всё, что не координация», + регулярный аудит логов.
11. **Конфликт «два агента в одной группе».** Если все 4 бота в одной supergroup, и `requireMention: false` — каждый бот видит каждое сообщение и может отвечать. Решение: `requireMention: true` для всех вторичных ботов в их `bindings`/`accounts.<id>` config.
12. **Mention-collision.** Если у research mention `@r`, а у пользователя есть слово «error» — может ложно сработать. Используй уникальные mention типа `@dim-r` или полные `@research`.
13. **A2A protocol — НЕ интегрирован в OpenClaw из коробки** [VERIFY]. Если хочешь связать OpenClaw-агента с внешним A2A-агентом (например, Microsoft) — нужен mediator-скилл, который говорит на A2A через HTTP+SSE+JSON-RPC.

## 🧪 Верификация

1. **Список агентов и bindings:**
   ```bash
   openclaw agents list --bindings
   ```
   Должны быть видны все 4 агента и их матчинги. Если binding для research нет — main всегда будет себе всё забирать.

2. **Health check каналов:**
   ```bash
   openclaw channels status --probe
   ```
   Все аккаунты — `OK`. `401` = битый токен (regenerate в BotFather).

3. **Sub-agent spawn smoke test (внутри сессии main):**
   ```
   /subagents spawn research "напиши 'pong' и заверши" --model claude-3-5-haiku-20241022
   /subagents list
   ```
   Через ~10 секунд — `Status: success`. Если timed out / failed — проверь `agentDir` research, ключ модели, runTimeoutSeconds.

4. **Push announcement в действии:**
   После завершения research — в твоей сессии main должен появиться announce. Если не появляется в течение `runTimeoutSeconds` — это баг push-механизма (best-effort).

5. **Нет cross-contamination workspace:**
   ```bash
   ls ~/.openclaw/workspace-research/
   ls ~/.openclaw/workspace-content/
   diff <(ls ~/.openclaw/workspace-research/) <(ls ~/.openclaw/workspace-content/)
   ```
   `SOUL.md` файлы должны различаться. Если идентичны — ты случайно скопировал, перезапиши.

6. **Telegram-роуты живые:**
   - DM `@dim_main_bot` → ответ от main.
   - В `#research` топике `@research найди X` → ответ от research, не от main.
   - Если в `#research` отвечает main — `bindings` приоритет неправильный, поправь порядок в массиве (peer/account match идёт раньше default).

7. **Бюджет:**
   После первого дня работы — `openclaw cost report` (если такая команда есть, иначе — анализ логов). Цель: main < 30% общих токенов, research/content/support — основная масса. Если main на Opus = 80% — он мало делегирует, ужесточи SOUL.md.

8. **Loki Mode reference [VERIFY]:** Это локальный системный скилл «multi-agent autonomous startup system». Полезен как идейный референс для авто-старта команды (cron создаёт sub-agents под задачи бэклога), но НЕ часть OpenClaw напрямую — нужно адаптировать.
