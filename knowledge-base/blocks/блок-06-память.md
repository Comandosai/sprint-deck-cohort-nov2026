## 🎯 Цель блока

Превратить агента из «амнезика, забывающего всё после `/reset`» в систему, которая:

1. **Помнит факты долгосрочно** — клиенты, проекты, ключевые решения, повторяющиеся задачи живут в `~/.openclaw/MEMORY.md` и тематических `memory/*.md`.
2. **Находит нужное за миллисекунды** — гибридный поиск (vectors + BM25 + temporal decay + MMR) возвращает топ-5 релевантных фрагментов, а не 200k токенов «всей памяти».
3. **Не раздувает контекст** — compaction срезает неважное при достижении `softThresholdTokens=40000`, contextPruning в режиме `cache-ttl` сохраняет prompt-cache.
4. **Самообучается из daily logs** — `memory/2026-04-29.md` пишется автоматически, на следующий день становится источником для `memory_search`.
5. **Фильтрует privacy** — пароли, API-ключи, BIN-коды никогда не попадают в файлы (regex-guard на запись).
6. **Проходит amnesia-test** — после `/reset` помнит, что «Иван — CTO Acme, бюджет $50k, дедлайн 15 мая».

## ⚡ Что нового в апреле 2026

| Новое | Что это даёт |
|---|---|
| **Hybrid retrieval по умолчанию** в большинстве агентских фреймворков (Claude Code memory tool, mem0 v0.2, Letta 0.6) — не «или vectors, или BM25», а оба с весами 0.6/0.4 | Лексический поиск ловит точные имена («Acme», SKU, артикул), векторный — семантику («наш крупный клиент») |
| **Temporal decay стал стандартом** — Zep представил его в community ещё в 2023, в 2026 его поддерживают все нормальные memory-системы | Старые факты не исчезают, но приоритет падает — «решение от 2024-01-15» не вытеснит «решение от 2026-04-20» |
| **MMR (λ=0.5–0.7)** против дублей в выдаче | Раньше top-5 могли быть пятью почти одинаковыми чанками; MMR вернёт один точный + четыре разнообразных |
| **Cohere Rerank v3 / bge-reranker-v2-m3** как пост-фильтр top-50 → top-5 | +15–25% precision@5 практически бесплатно (~50ms) |
| **Структурированные факты (JSON-frontmatter в .md)** вытесняют свободный текст | LLM хуже извлекает «Иван (CTO, Acme, ivan@acme.com)» из абзаца, чем из YAML; embedding качество выше |
| **Knowledge graph как опция** (Cognee, Graphiti) | Для блока 20 — связи «Иван → работает в → Acme → клиент с → 2024-03» дают 30%+ к точности на multi-hop вопросах |
| **prompt cache-aware contextPruning** (Anthropic явно описал в context-engineering doc 2025) | Не вырезай начало prompt — оно закешировано; режь хвост / середину |
| **«Forgetting curve» политика** — Zep, mem0, Letta поддерживают TTL на факты | Не всё хранить вечно: PII со старых проектов истекает через 180 дней автоматом |

## 💡 Лайфхаки и про-приёмы

### 1. Структурированные факты с YAML frontmatter, а не свободный текст
Дай каждому факту метаданные — `type`, `tags`, `confidence`, `source`, `validUntil`. Embedding такого факта точнее, фильтрация по `tags=client` работает мгновенно, retrieval через MMR не дублирует.

```yaml
---
type: fact
entity: Иван Петров
role: CTO
company: Acme Corp
confidence: 0.95
source: meeting-2026-04-15
validUntil: 2026-12-31
tags: [client, acme, contact]
---
```

### 2. Правило «500 строк или 3 темы» для выделения в отдельный файл
Не выноси в `memory/X.md` каждый чих. Критерии:
- MEMORY.md разрастается > 500–700 строк → пора резать
- Тема имеет ≥ 3 разных подтемы (Acme → контакты + проекты + договоры)
- Тема обновляется чаще 1 раз в неделю
- Тема имеет жизненный цикл (клиент закрылся — архивируем `memory/archive/client-acme.md`)

### 3. Hybrid weights: начинай с 0.6 vectors / 0.4 BM25, тюнь по логу
Чисто vectors путает синонимы («Acme» с «Acme Inc»), чисто BM25 не видит «крупный клиент» = «Acme». Тюнинг: возьми 30 реальных запросов из логов за неделю, прогоняй с разными weights, считай recall@5.

### 4. Temporal decay halfLifeDays — три профиля
- **7 дней** — оперативка («что обсуждали на этой неделе»), daily logs
- **30 дней** — стандарт для большинства фактов о проектах в работе
- **90 дней** — стабильные факты о клиентах (контакты, реквизиты)
- **365+** — «вечные» факты (имена, компании, ключевые решения проекта)
Не ставь decay на personal/USER.md — там вообще не должно быть decay.

### 5. MMR λ: 0.5 для исследования, 0.7 для точности
- λ=0.7 (ближе к чистой релевантности) — когда нужен конкретный факт («какой email Ивана»)
- λ=0.5 (сильнее diversity) — когда нужен обзор («что мы знаем про Acme»)
Контролируй через тип запроса: regex `\b(какой|какая|какие|что такое)\b` → λ=0.7, иначе 0.5.

### 6. Compaction срезает middle, а не head/tail
Голова системного промпта закеширована (prompt cache TTL 5 мин), хвост — самое свежее и самое важное. Compaction должен вырезать middle: суммаризировать 5–20 сообщений из середины в один абзац-конспект. Для prompt cache это критично — сэкономит 50–90% стоимости.

### 7. Daily logs — append-only, со схемой
Не давай агенту переписывать старые логи. Шаблон фиксирован, новое всегда добавляется в конец. На следующий день это становится источником для retrieval. Раз в неделю — задача `memory:weekly-digest` запускает суммаризацию 7 daily-logs в один абзац для долгой памяти.

### 8. Privacy guard на запись (write-time, не read-time)
Если регулярка ловит пароль/токен/BIN-карту в content — БЛОКИРУЙ запись с ошибкой, а не «маскируй». Маскированный пароль `pa****rd` всё ещё может слить инфо. Список патернов: `detect-secrets` plugins + кастом для рос. реалий (СНИЛС `\d{3}-\d{3}-\d{3} \d{2}`, ИНН `\d{10,12}`).

### 9. Cross-session memory через единый storage path
Telegram-сессия, email-сессия, CLI — все указывают на один `~/.openclaw/MEMORY.md`. Не дублируй. Для production — одна общая база (LanceDB на S3, или pgvector на managed Postgres). Если будут конфликты записи — `memory_lock` через файл-семафор или advisory lock в Postgres.

### 10. Конфликт фактов: версии + confidence
Не перезаписывай. «Иван работает в Acme (conf 0.95, 2024-03)» + «Иван работает в Beta (conf 0.95, 2026-04-20)» — это два факта. Retrieval вернёт оба, агент в системном промпте инструктируется: «при противоречии — выбирай факт с более свежим `source.date`». Это решает 90% случаев перехода клиента.

### 11. Source citing — каждый факт с `source: meeting-2026-04-15` или ссылкой на daily log
Без этого ты не разберёшь, откуда галлюцинация. С этим — `grep -r "ivan@acme.com" ~/.openclaw/memory/` мгновенно показывает первоисточник.

### 12. Forgetting strategy: TTL + tombstones
Не удаляй факты hard-delete. Помечай `archived: true` в frontmatter — retrieval их игнорит, но они остаются в git history и в `memory/archive/`. TTL: PII клиента → 180 дней после закрытия проекта. Daily logs → 90 дней онлайн, потом `tar.gz` в архив.

### 13. Amnesia-test как CI-job
Не «протестировал руками один раз». Положи в репо `tests/amnesia.sh`, гоняй еженедельно cron-job. Скрипт: вкидывает 5 фактов → `/reset` → задаёт 5 вопросов → проверяет ответы regex-ом → пишет в Slack если recall < 0.8.

### 14. Reranking — only for top-N, а не каждый запрос
Cohere Rerank стоит денег. Применяй ТОЛЬКО когда:
- Hybrid вернул > 20 кандидатов (часто это значит запрос двусмысленный)
- Запрос в категории «critical» (помечается тегом `priority:high`)
- Long-context: нужно сжать 50 → 5 для дорогих сценариев

### 15. Git diff как «changelog памяти»
В блоке 14 настроен git push для `~/.openclaw/`. Раз в неделю запускай `git log --since="1 week ago" --diff-filter=A,M -- 'memory/*.md'` — увидишь, что агент добавил/изменил. Странности (внезапно новый клиент, которого ты не знаешь) → разбирайся.

### Bash-скрипт `tests/amnesia.sh` (amnesia-test)

```bash
#!/usr/bin/env bash
# tests/amnesia.sh — проверяет, что агент помнит факты после /reset
set -euo pipefail

OPENCLAW="${OPENCLAW_BIN:-openclaw}"
LOG="/tmp/amnesia-$(date +%s).log"

pass=0
fail=0

ask() {
  local prompt="$1"
  local pattern="$2"
  local label="$3"
  local out
  out="$($OPENCLAW chat --no-stream "$prompt" 2>&1 | tail -50)"
  echo "=== $label ===" >> "$LOG"
  echo "$out" >> "$LOG"
  if grep -Eqi "$pattern" <<<"$out"; then
    echo "PASS  $label"
    ((pass++))
  else
    echo "FAIL  $label  (expected /$pattern/)"
    ((fail++))
  fi
}

echo "[1/3] Сообщаем агенту факты..."
$OPENCLAW chat --no-stream \
  "Запомни: Иван Петров — CTO Acme Corp, ivan@acme.com. \
   Бюджет проекта Acme KB — \$30,000. Дедлайн MVP — 15 мая 2026. \
   Мария Сидорова — Head of Product, не решает бюджет." > /dev/null

echo "[2/3] Сбрасываем сессию..."
$OPENCLAW reset --confirm

echo "[3/3] Спрашиваем после reset:"
ask "Кто CTO в Acme?"                 "Иван\s+Петров"             "recall: имя CTO"
ask "Какой email у Ивана из Acme?"    "ivan@acme\.com"            "recall: email"
ask "Какой бюджет проекта Acme KB?"   "30[\s\.,]?000|\\\$30k"      "recall: бюджет"
ask "Когда дедлайн MVP?"              "15.*мая|2026-05-15"         "recall: дедлайн"
ask "Кто решает бюджет в Acme?"       "не\s+Мария|Иван|CTO"        "recall: ответственность"

total=$((pass+fail))
recall=$(awk -v p=$pass -v t=$total 'BEGIN { printf "%.0f", (p/t)*100 }')

echo "----"
echo "Recall: $recall% ($pass/$total)"
echo "Лог: $LOG"

if [ "$recall" -lt 80 ]; then
  echo "ERROR: recall ниже 80% — память не работает корректно"
  exit 1
fi
```

## ⚠️ Подводные камни

| Подводный камень | Симптом | Что делать |
|---|---|---|
| **`text-embedding-3-small` слабее на русском, чем на английском** | Recall на русских запросах ~70%, на английских ~88% | Для серьёзного русского — `voyage-3` или `bge-m3` (multilingual). Или дублируй ключевые факты на EN в `aliases:` frontmatter |
| **Compaction съедает важные факты** | Через неделю агент «забывает» решение, которое было в чате | Помечай критические сообщения `tag:decision` и добавь `preserveTags: [decision]` в compaction config |
| **MMR с λ=0.3 даёт «обзор», но теряет точные совпадения** | На вопрос «email Ивана» возвращается список людей вместо ответа | Динамический λ: для вопросов с явным entity → 0.7+, для общих → 0.5 |
| **Cache-TTL 5 мин не совпадает с rate limit** | Cache miss каждый раз → дорого | Согласуй с реальным TTL Anthropic prompt cache (5 мин) и держи активные сессии плотно |
| **LanceDB файл растёт линейно без `optimize()`** | Через 3 месяца поиск замедлился в 4–8 раз | Cron `lancedb optimize` ежемесячно + `compact` после массовых удалений |
| **Privacy-guard на чтение, а не на запись** | Секрет уже в файле, маскируется при выдаче, но просочился в git | Жёстко: блокируй на write. Если уже просочился — `git filter-repo` + ротация секрета |
| **Конфликт фактов разрешается «последний выигрывает»** | Ошибочное обновление перетёрло корректный факт | Включи `useConfidence: true` + `minConfidenceDelta: 0.1` — без явно более высокой уверенности новый факт идёт в историю, не перезаписывает |
| **Daily log переписывается, а не append** | Утренние записи исчезают к вечеру | `appendOnly: true` обязательно. Проверяй ACL: chmod 644, владелец — пользователь openclaw |
| **MEMORY.md > 30k токенов целиком в контексте** | Каждый запрос дорогой; cache hit низкий | Не подгружай весь MEMORY.md в prompt. Только индекс (раздел 0) + retrieval top-5. Полный файл — только при явной команде |
| **Reranker timeout срывает ответ** | Запрос висит 30 сек, потом падает | `timeoutMs: 800` + fallback на hybrid без rerank. Reranker — не обязателен, а enhancer |
| **Rотация daily logs ломает retrieval** | После архивации 90+ дней назад не находит старый факт | Перед `tar.gz` запускай `weeklyDigest`, чтобы суммари осталось онлайн |
| **Embedding не пересчитывается при правке файла** | Файл изменён, поиск возвращает старую версию | Watcher на `~/.openclaw/memory/**` (chokidar/inotify) → ре-эмбеддинг по mtime |
