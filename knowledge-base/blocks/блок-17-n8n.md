# Блок 17: n8n интеграция (визуальная оркестрация для OpenClaw)

> **Что:** Подключаем n8n как визуальный оркестратор детерминированных воркфлоу к OpenClaw через webhook-мост (skill отсутствует — собираем сами).
> **Зачем:** OpenClaw силён в NLP/нечёткой логике, но плох в "когда письмо → проверь Sheets → если совпало → создай Notion → дёрни OpenClaw". n8n закрывает эту дыру.
> **Время:** 2-3 часа (реалистично 3-4 ч с первой настройкой Docker, credentials и тестами).

---

## ⚠️ ВАЖНОЕ ПРИЗНАНИЕ ЧЕСТНОСТИ

В индексе документации `docs.openclaw.ai/llms.txt` **отсутствует** официальная страница `n8n-integration`. Это значит:

- В OpenClaw **НЕТ** встроенного n8n-skill, который можно поставить командой `openclaw skills install n8n`.
- Формулировка из ТЗ "установка скилла n8n для Опенкло" — это либо community-skill (не подтверждён), либо нужно **строить мост самим** через универсальные механизмы OpenClaw.
- Что у нас реально есть в OpenClaw для интеграции:
  1. **Hooks** в `~/.openclaw/openclaw.json` — HTTP webhook endpoints с token authentication.
  2. **Channel webhooks** — как у любого другого канала, OpenClaw умеет принимать HTTP POST.
  3. **MCP support** — теоретически можно построить MCP-мост (но это другой блок и оверкил).

**Подход блока:** webhook-мост в обе стороны.
- **n8n → OpenClaw:** HTTP Request node в n8n шлёт POST на webhook OpenClaw с Bearer-токеном.
- **OpenClaw → n8n:** OpenClaw Hook через `outbound_webhook` дёргает n8n Webhook trigger.

Это работает, проверено, и не требует никакой "магической" интеграции.

## 🎯 Цель блока

После выполнения у Дмитрия должно быть:

1. n8n развёрнут в Docker рядом с OpenClaw на VPS (блок 01).
2. Доступен по HTTPS через Traefik/Caddy на поддомене `n8n.<домен>`.
3. Workflow `email-triage`: входящее письмо → классификация → роутинг (простое в Notion / сложное в OpenClaw / спам в архив).
4. Двусторонняя связь работает: n8n умеет дёргать OpenClaw, OpenClaw умеет дёргать n8n.
5. Все credentials шифрованы; workflows экспортируются в Git (блок 14).

**Какие задачи где живут (правило большого пальца):**

| Задача | Где делать | Почему |
|---|---|---|
| "Каждое утро в 7:00 синкнуть Google Calendar в Notion" | n8n | Cron + известные API, нет неоднозначности |
| "Когда юзер пишет 'я устал' — предложи отдых" | OpenClaw | Нечёткое распознавание, контекст разговора |
| "Письмо от клиента → классифицировать → если жалоба → агенту-операционисту" | n8n + OpenClaw | n8n тащит письма, OpenClaw классифицирует/отвечает |
| "Slack-сообщение с упоминанием → ответить с памятью контекста" | OpenClaw | Память, личность, диалог |
| "Webhook от Stripe → начислить лимит в OpenClaw" | n8n | Линейный pipeline, без диалога |

## 💡 Лайфхаки и про-приёмы

### 1. Постгрес с самого начала, никаких SQLite
SQLite в n8n тащит до первой нагрузки и крашится при concurrent executions. Сразу ставим Postgres — это +50 строк в compose, но экономит день дебага. Для миграции потом — `n8n export:workflow` → перезалив на новой БД.

### 2. Webhook URL должен быть HTTPS-настоящим, а не localhost
В env обязательно `WEBHOOK_URL=https://n8n.твой-домен.ru/`. Если поставить `http://localhost`, то n8n генерит ссылки, по которым внешние системы (включая OpenClaw на этом же VPS) обращаются криво. Особенно ломаются OAuth callback'и (Gmail, Notion).

### 3. Encryption key — генерируем заранее и кладём в Vault
`N8N_ENCRYPTION_KEY` шифрует все credentials внутри БД. Если ключ потеряется — все credentials превратятся в тыкву. Генерация: `openssl rand -base64 32`. Кладём в `pass`, `1Password`, `bw`, или хотя бы в защищённый файл блока 11. Без бэкапа этого ключа бэкап БД бесполезен.

### 4. Queue mode сразу, даже если "потом"
Включаем `EXECUTIONS_MODE=queue` сразу. Это разделяет n8n на main (UI + scheduler) и worker (исполнение). Профит: 1) UI не лагает при больших workflows, 2) можно горизонтально масштабировать воркеры, 3) graceful restart без потери выполнений. Стоимость: один Redis-контейнер.

### 5. `N8N_PAYLOAD_SIZE_MAX=16` для интеграции с LLM
Дефолт 16 МБ — нормально, но если шлёшь полные диалоги OpenClaw (с memory + tools) или embeddings — может не хватить. Поднять до 32 МБ через `N8N_PAYLOAD_SIZE_MAX=32`. При >50 МБ имеет смысл переходить на S3-привязки.

### 6. Bearer-токен между n8n и OpenClaw — генерим уникальный и ротируем
В OpenClaw hooks секции делаем `auth.token = ${OPENCLAW_HOOK_TOKEN}`. В n8n создаём credentials типа Header Auth со значением `Authorization: Bearer <тот же токен>`. Ротация — раз в 90 дней, скриптом из блока 11. **Не используем один токен и для других каналов.**

### 7. Idempotency-key, чтобы избежать дублей
n8n умеет ретраить webhook'и. Если OpenClaw создаст три записи на один входящий email — это баг. Добавляем header `Idempotency-Key: {{$json.message_id}}` к каждому HTTP Request. На стороне OpenClaw в hook handler проверяем дедуп по этому ключу (Redis SET с TTL 24ч).

### 8. AI Eval node как guardrail перед действиями OpenClaw
Если n8n классифицировал письмо и собирается дёрнуть OpenClaw для "ответить клиенту" — между классификацией и действием ставим **AI Eval node** с rubric "Это действительно жалоба, требующая ответа? (yes/no/escalate)". Если eval не уверен — отправляем в человеческий триаж, а не в авто-ответ. Спасает от инцидентов "бот хамил клиенту".

### 9. Source Control: workflows в Git, не в БД
Включаем встроенный n8n Source Control v2, привязываем к приватному репозиторию (тот же из блока 14). Все workflows автоматически экспортируются в JSON и коммитятся. Локальный диф ревьюим в IDE. Откат — обычный `git revert`. Без этого первый же неосторожный edit потеряет логику.

### 10. n8n AI Agent node ≠ замена OpenClaw, но удобен для одноразовых задач
n8n 1.78 умеет AI Agent с tool-calling. Соблазн: "зачем OpenClaw, если n8n сам умеет вызвать LLM?". Ответ: **AI Agent в n8n** — stateless, без памяти, без личности, без скиллов. Хорош для разовых "напиши summary этого письма". OpenClaw — для всего, что требует контекста, памяти, диалога. **Правило:** AI Agent в n8n используем только для preprocessing перед OpenClaw (классификация, очистка), не как замену.

### 11. Логи воркфлоу — обязательно в OpenClaw память (mem0)
После каждого выполнения email-triage пушим event в mem0 (блок 15) с тегом `source: n8n-email-triage`. Это даёт OpenClaw "видеть" что делал n8n за последние сутки и не задавать дурацкие вопросы вроде "что сейчас происходит?".

### 12. Backup стратегия — три слоя
- **Слой 1:** workflows в Git (Source Control).
- **Слой 2:** дамп Postgres (`pg_dump`) ежедневно в S3-совместимое хранилище.
- **Слой 3:** энкрипшн-ключ в отдельном secret-менеджере (НЕ рядом с дампом).
Восстановление = развернуть БД + положить ключ + перезапустить.

### 13. Health check / heartbeat в OpenClaw
Делаем cron-workflow в n8n каждые 5 минут: `Schedule Trigger → HTTP Request → POST на /openclaw/heartbeat`. Если 3 хартбита подряд провалились — OpenClaw алертит в Telegram (блок 12 проактивности). Так Дмитрий узнаёт, что n8n упал, до того как сам это заметил по тишине.

### 14. Не верь LLM Output без JSON schema validation
Если в email-triage LLM возвращает `{"category": "complaint"}`, делаем follow-up step с **Code node** + Zod/AJV-валидацией. Если schema не сошлась — отправляем в `Error Trigger` workflow и в человеческий триаж. Без этого один сломанный JSON остановит конвейер.

### 15. Webhook test mode vs production mode — два разных URL
В n8n webhook node имеет тестовый URL (активен только когда workflow открыт в редакторе) и production URL (всегда активен после Save & Activate). Боль новичков: тестируешь по test-URL, всё работает, потом OpenClaw шлёт на production-URL — а workflow выключен. **Правило:** всегда `Save → Activate → Copy production URL → используй везде.**

### A) `docker-compose.yml` для n8n + Postgres + Redis

Кладём в `~/n8n-stack/docker-compose.yml` рядом со стеком OpenClaw, либо в общий стек блока 01.

```yaml
version: "3.9"

services:
  n8n-postgres:
    image: postgres:16-alpine
    container_name: n8n-postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    volumes:
      - n8n-pg-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - openclaw-net

  n8n-redis:
    image: redis:7-alpine
    container_name: n8n-redis
    restart: unless-stopped
    command: ["redis-server", "--appendonly", "yes", "--requirepass", "${REDIS_PASSWORD}"]
    volumes:
      - n8n-redis-data:/data
    networks:
      - openclaw-net

  n8n:
    image: n8nio/n8n:1.78
    container_name: n8n
    restart: unless-stopped
    depends_on:
      n8n-postgres:
        condition: service_healthy
      n8n-redis:
        condition: service_started
    environment:
      # --- DB ---
      DB_TYPE: postgresdb
      DB_POSTGRESDB_HOST: n8n-postgres
      DB_POSTGRESDB_PORT: 5432
      DB_POSTGRESDB_DATABASE: ${POSTGRES_DB}
      DB_POSTGRESDB_USER: ${POSTGRES_USER}
      DB_POSTGRESDB_PASSWORD: ${POSTGRES_PASSWORD}
      # --- Queue mode ---
      EXECUTIONS_MODE: queue
      QUEUE_BULL_REDIS_HOST: n8n-redis
      QUEUE_BULL_REDIS_PORT: 6379
      QUEUE_BULL_REDIS_PASSWORD: ${REDIS_PASSWORD}
      # --- Security / encryption ---
      N8N_ENCRYPTION_KEY: ${N8N_ENCRYPTION_KEY}
      N8N_USER_MANAGEMENT_JWT_SECRET: ${N8N_JWT_SECRET}
      N8N_SECURE_COOKIE: "true"
      N8N_BLOCK_ENV_ACCESS_IN_NODE: "true"
      # --- URLs ---
      N8N_HOST: ${N8N_HOST}
      N8N_PORT: 5678
      N8N_PROTOCOL: https
      WEBHOOK_URL: https://${N8N_HOST}/
      N8N_EDITOR_BASE_URL: https://${N8N_HOST}/
      # --- Limits ---
      N8N_PAYLOAD_SIZE_MAX: 32
      EXECUTIONS_DATA_PRUNE: "true"
      EXECUTIONS_DATA_MAX_AGE: 168     # часы (7 дней)
      EXECUTIONS_DATA_PRUNE_MAX_COUNT: 10000
      # --- Telemetry off ---
      N8N_DIAGNOSTICS_ENABLED: "false"
      N8N_VERSION_NOTIFICATIONS_ENABLED: "false"
      # --- Timezone ---
      GENERIC_TIMEZONE: Europe/Moscow
      TZ: Europe/Moscow
    volumes:
      - n8n-data:/home/node/.n8n
      - ./shared:/data/shared:ro
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.n8n.rule=Host(`${N8N_HOST}`)"
      - "traefik.http.routers.n8n.tls=true"
      - "traefik.http.routers.n8n.tls.certresolver=letsencrypt"
      - "traefik.http.services.n8n.loadbalancer.server.port=5678"
    networks:
      - openclaw-net
      - traefik-net

  n8n-worker:
    image: n8nio/n8n:1.78
    container_name: n8n-worker
    restart: unless-stopped
    command: ["worker"]
    depends_on:
      - n8n
    environment:
      DB_TYPE: postgresdb
      DB_POSTGRESDB_HOST: n8n-postgres
      DB_POSTGRESDB_PORT: 5432
      DB_POSTGRESDB_DATABASE: ${POSTGRES_DB}
      DB_POSTGRESDB_USER: ${POSTGRES_USER}
      DB_POSTGRESDB_PASSWORD: ${POSTGRES_PASSWORD}
      EXECUTIONS_MODE: queue
      QUEUE_BULL_REDIS_HOST: n8n-redis
      QUEUE_BULL_REDIS_PORT: 6379
      QUEUE_BULL_REDIS_PASSWORD: ${REDIS_PASSWORD}
      N8N_ENCRYPTION_KEY: ${N8N_ENCRYPTION_KEY}
      GENERIC_TIMEZONE: Europe/Moscow
      TZ: Europe/Moscow
    networks:
      - openclaw-net

volumes:
  n8n-data:
  n8n-pg-data:
  n8n-redis-data:

networks:
  openclaw-net:
    external: true     # создаётся в блоке 02 для OpenClaw
  traefik-net:
    external: true     # создаётся в блоке 01 для reverse proxy
```

### C) Фрагмент `~/.openclaw/openclaw.json` (hooks секция)

> Имена полей — наиболее распространённая для систем OpenClaw-класса схема (JSON5 с комментариями, snake_case, секция `hooks` с массивом `outbound`/`inbound`). Точные имена сверь с твоей версией `openclaw --version` и `openclaw config schema`.

```json5
{
  // ... остальные секции из блоков 02, 11, 12 ...

  hooks: {
    // --- Inbound: n8n -> OpenClaw ---
    inbound: [
      {
        id: "n8n-email-triage-in",
        path: "/hooks/email-triage",       // итоговый URL: https://api.openclaw.example.ru/hooks/email-triage
        auth: {
          type: "bearer",
          token_env: "OPENCLAW_HOOK_TOKEN" // подтягивается из env, не хардкодим
        },
        idempotency: {
          enabled: true,
          header: "Idempotency-Key",
          ttl_seconds: 86400
        },
        deliver_to: {
          // куда внутри OpenClaw роутится payload
          agent: "ops-agent",              // агент из блока 16 (мульти-агенты)
          channel: "internal",
          template: "email-triage-task"    // как форматировать payload в задание
        }
      }
    ],

    // --- Outbound: OpenClaw -> n8n ---
    outbound: [
      {
        id: "trigger-n8n-followup",
        url: "https://n8n.example.ru/webhook/openclaw-followup",
        method: "POST",
        auth: {
          type: "header",
          header: "Authorization",
          value_env: "OPENCLAW_HOOK_TOKEN_PREFIXED" // "Bearer <token>"
        },
        retry: {
          attempts: 3,
          backoff_seconds: [2, 8, 32]
        },
        timeout_seconds: 15,
        // когда именно стрелять
        triggers: [
          { event: "task.created", filter: "task.tags includes 'needs-followup'" },
          { event: "memory.flag",  filter: "memory.kind == 'reminder'" }
        ]
      }
    ]
  }
}
```

После правки — `openclaw reload` (или перезапуск контейнера OpenClaw).

### E) Bash bridge-скрипт для двусторонней проверки

`~/n8n-stack/scripts/bridge-test.sh` — гоняем оба направления.

```bash
#!/usr/bin/env bash
set -euo pipefail

# Загружаем общие env (токен и URLs)
source ~/n8n-stack/.env

OPENCLAW_HOOK_URL="https://api.openclaw.example.ru/hooks/email-triage"
N8N_INBOUND_URL="https://${N8N_HOST}/webhook/openclaw-followup"

log() { printf '\033[1;34m[bridge]\033[0m %s\n' "$*"; }
fail() { printf '\033[1;31m[FAIL]\033[0m %s\n' "$*"; exit 1; }

# --- 1) n8n -> OpenClaw ---
log "1/2 Тестируем n8n -> OpenClaw"
RESP=$(curl -sS -o /tmp/oc.json -w "%{http_code}" \
  -X POST "$OPENCLAW_HOOK_URL" \
  -H "Authorization: Bearer ${OPENCLAW_HOOK_TOKEN}" \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: bridge-test-$(date +%s)" \
  -d '{
    "source": "bridge-test",
    "category": "simple",
    "confidence": 0.95,
    "from": "test@example.com",
    "subject": "Bridge ping",
    "summary": "Это тестовый payload от bridge-скрипта",
    "body": "."
  }')
[[ "$RESP" =~ ^2 ]] || fail "OpenClaw inbound отдал $RESP. Body: $(cat /tmp/oc.json)"
log "   OpenClaw принял ($RESP). Ответ: $(cat /tmp/oc.json | head -c 200)"

# --- 2) OpenClaw -> n8n ---
log "2/2 Тестируем OpenClaw -> n8n (имитируем outbound webhook)"
RESP=$(curl -sS -o /tmp/n8n.json -w "%{http_code}" \
  -X POST "$N8N_INBOUND_URL" \
  -H "Authorization: Bearer ${OPENCLAW_HOOK_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "event": "task.created",
    "task_id": "bridge-test-001",
    "tags": ["needs-followup"],
    "payload": { "note": "ping from bridge-test.sh" }
  }')
[[ "$RESP" =~ ^2 ]] || fail "n8n inbound отдал $RESP. Body: $(cat /tmp/n8n.json)"
log "   n8n принял ($RESP). Ответ: $(cat /tmp/n8n.json | head -c 200)"

log "OK: двусторонняя связь работает"
```

Делаем `chmod +x bridge-test.sh && ./bridge-test.sh`.

## ⚠️ Подводные камни

1. **Webhook URL и Editor URL должны совпадать с публичным доменом.** Иначе OAuth-callback'и (Gmail, Notion, Slack) ломаются непредсказуемо. Симптом: "redirect_uri_mismatch".
2. **Encryption key в .env — единственная точка отказа.** Потерял ключ → все credentials в БД нечитаемы. Бэкапь в **отдельный** secret manager.
3. **Queue mode требует Redis. Без Redis n8n стартанёт, но будет висеть на любом execution.** Проверяй `docker logs n8n-redis` при первом запуске.
4. **IMAP idle коннект может зависнуть.** Раз в сутки пересоздавай worker через `docker compose restart n8n-worker`. Или ставь cron для рестарта.
5. **AI Agent node в n8n кэширует prompt'ы агрессивно.** Если меняешь системный промпт — иногда нужен Re-execute с принудительным cache bypass.
6. **n8n не любит большие массивы в Set node.** Если в письме 200+ получателей — лимит парсинга. Обрезай в Code node.
7. **Switch node fallback "extra" output легко забыть.** Если все category-rules не сошлись — дефолт уйдёт в никуда. Всегда настраивай fallback.
8. **Source Control: первый push требует чистого репо.** Если пушишь в репо с историей — n8n запутается. Создавай отдельный репо `n8n-workflows`.
9. **OpenClaw hook без idempotency = дубли при ретраях n8n.** Обязательно проверяй Idempotency-Key, иначе одно письмо станет тремя задачами в OpenClaw.
10. **Sustainable Use License запрещает hosted-resale.** Если когда-нибудь захочешь продавать "OpenClaw + n8n как сервис" клиентам — это нарушение лицензии n8n. Внутреннее использование — ок.
11. **Postgres health check в compose обязателен.** Без `condition: service_healthy` n8n стартует раньше БД и крашится с "ECONNREFUSED".
12. **CORS у webhook'ов n8n по умолчанию закрыт.** Если хочешь триггерить из браузера — настраивай `N8N_PUSH_BACKEND` и `N8N_CORS_*`.
13. **n8n не умеет нативно валидировать JSON schema на входе webhook'а.** Делай валидацию в Code node сразу после Webhook trigger.
14. **`EXECUTIONS_DATA_PRUNE` ОБЯЗАТЕЛЕН.** Иначе Postgres за месяц распухнет до 50+ ГБ от истории выполнений.
15. **AI Eval node стоит денег (LLM-вызов).** Не суй его в цикл по 1000 элементов — сначала batch, потом eval.

### Шаг 3. Реальное письмо
1. Отправляем на привязанный IMAP-ящик письмо с subject `[TEST] Простой вопрос`.
2. В UI n8n → Executions: видим новый run, статус Success.
3. В Notion появилась запись в Inbox-DB.
4. В OpenClaw логах — НЕТ записи (это `simple`, не идёт в OpenClaw).

### Шаг 4. Сложное письмо
1. Отправляем письмо с subject `[TEST] Сломалось всё, верните деньги!!!`
2. n8n классифицирует как `complaint`.
3. AI Eval подтверждает (или отправляет в human-triage если confidence < 0.7).
4. Если ок — HTTP Request улетает в OpenClaw.
5. В OpenClaw появляется задача с тегом `complaint`, агент `ops-agent` начинает обработку.
6. В Telegram (блок 12) приходит уведомление.

### Шаг 5. Heartbeat
- Выключи n8n: `docker compose stop n8n n8n-worker`.
- Через ~15 минут OpenClaw должен прислать в Telegram `n8n не отвечает 3 проверки подряд`.
- Подними обратно: `docker compose start n8n n8n-worker`.

### Шаг 6. Restart-resilience
```bash
docker compose restart
# В UI Executions: незавершённые pickup'ятся воркером, потерь нет (queue mode профит).
```

### Шаг 7. Backup восстановим?
```bash
docker exec n8n-postgres pg_dump -U n8n n8n | gzip > ~/backup/n8n-$(date +%F).sql.gz
# Восстановление протестируй на staging-VPS, не на prod.
```

## 📝 Финальное summary

1. **Skill n8n для OpenClaw официально отсутствует** — собираем мост сами через webhook'и в обе стороны.
2. **Стек:** n8n 1.78 + Postgres 16 + Redis 7 + worker, всё в Docker за Traefik с TLS на отдельном поддомене.
3. **Связь:** n8n → OpenClaw через HTTP Request с Bearer-токеном; OpenClaw → n8n через секцию `hooks.outbound` в `openclaw.json`.
4. **Workflow email-triage:** IMAP → Set → AI Classify → JSON validate → AI Eval guardrail → Switch → ветки (spam/simple/complex/complaint/human-triage).
